2. Pick a path

Recommended

A. App Platform (PaaS)

DigitalOcean's managed app hosting. You point it at a GitHub branch, it builds and deploys automatically. Zero server management.

  • ✅ Auto-deploy on git push
  • ✅ HTTPS, scaling, logs included
  • ✅ Easy rollback (one click)
  • ⚠ Slightly pricier (~$25–40/mo)
  • ⚠ Less low-level control

Best for: single-school deployments, MVPs, small teams.

Power-user

B. Droplet + Docker Compose

One $12–24/mo VM with Docker running all services. Manual but cheaper and more flexible.

  • ✅ Cheap (~$15–25/mo all-in)
  • ✅ Full SSH access, custom CRON jobs
  • ✅ Same setup runs on AWS / Hetzner if you ever switch
  • ⚠ You own backups + OS updates
  • ⚠ Deploy is SSH + git pull (or scripted)

Best for: cost-sensitive deployments, multiple schools, technical comfort.

3. Prerequisites (both paths)

  • DigitalOcean account — sign up at digitalocean.com, add a payment method.
  • GitHub repository — your code already lives there (frontend + backend in one repo or separate, both work).
  • Domain name — e.g. ecoleerp.in. Buy from Namecheap / GoDaddy / Cloudflare.
  • Cloudflare account (free) — for DNS + auto-SSL.
  • Cloudinary account (free tier) — already integrated; or use Spaces if you prefer.
  • Local prep — make sure backend compiles (yarn build) and the seed runs locally.

4. Path A — App Platform (Recommended)

4.1 Provision managed Postgres

1

Create a Postgres cluster

Cloud control panel → Databases → Create Database Cluster → PostgreSQL 15 → smallest plan (1 GB RAM, $15/mo) → same region you'll use for the app (e.g. Bangalore for India).

2

Grab the connection string

Connection Details → "Connection String" → copy. Looks like:

postgresql://doadmin:<PASSWORD>@db-postgresql-blr1-xxxxx.b.db.ondigitalocean.com:25060/defaultdb?sslmode=require
3

Allow inbound from App Platform

Cluster → Trusted Sources → Add → pick "App Platform" (DO knows its own egress IPs). No firewall headache.

4.2 (Optional) Provision Redis

You can skip Redis for v1 — the app falls back gracefully. Add it when you need WebSocket fan-out across multiple instances or strict rate limiting. 

Databases → Create → Redis 7 → smallest plan ($15/mo) → same region.
Connection string format: rediss://default:<PASSWORD>@<host>:25061

5. Path B — Droplet with Docker Compose

5.1 Create the Droplet

Droplets → Create → Ubuntu 24.04 → Premium Intel → 2 GB RAM ($12/mo). Pick Bangalore region. Add your SSH key. Enable backups (+20% = $2.40/mo) — strongly recommended.

5.2 First-boot setup (run as root, then create a sudo user)

# SSH in
ssh root@<droplet-ip>

# Update
apt update && apt upgrade -y

# Create non-root user
adduser ecole
usermod -aG sudo ecole
rsync --archive --chown=ecole:ecole ~/.ssh /home/ecole/

# Lock down SSH (optional but recommended)
sed -i 's/^#*PermitRootLogin .*/PermitRootLogin no/' /etc/ssh/sshd_config
systemctl restart ssh

# Install Docker
curl -fsSL https://get.docker.com | sh
usermod -aG docker ecole

# Install fail2ban + ufw
apt install -y fail2ban ufw
ufw allow OpenSSH
ufw allow 80
ufw allow 443
ufw --force enable

5.3 Drop in a docker-compose.yml

SSH as ecole → mkdir ~/erp && cd ~/erp → create docker-compose.yml:

services:
  postgres:
    image: postgres:15-alpine
    restart: always
    environment:
      POSTGRES_USER: erp_user
      POSTGRES_PASSWORD: ${POSTGRES_PASSWORD}
      POSTGRES_DB: school_erp
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U erp_user"]
      interval: 10s

  redis:
    image: redis:7-alpine
    restart: always

  api:
    image: ghcr.io/your-org/school-erp-api:latest
    restart: always
    depends_on:
      postgres: { condition: service_healthy }
    environment:
      NODE_ENV: production
      DATABASE_URL: postgresql://erp_user:${POSTGRES_PASSWORD}@postgres:5432/school_erp
      REDIS_URL: redis://redis:6379
      JWT_ACCESS_SECRET: ${JWT_ACCESS_SECRET}
      JWT_REFRESH_SECRET: ${JWT_REFRESH_SECRET}
      STORAGE_DRIVER: cloudinary
      CLOUDINARY_CLOUD_NAME: ${CLOUDINARY_CLOUD_NAME}
      CLOUDINARY_API_KEY: ${CLOUDINARY_API_KEY}
      CLOUDINARY_API_SECRET: ${CLOUDINARY_API_SECRET}
    ports:
      - "127.0.0.1:3000:3000"

  web:
    image: ghcr.io/your-org/school-erp-web:latest
    restart: always
    depends_on: [api]
    environment:
      NEXT_PUBLIC_API_URL: https://api.yourschool.com
    ports:
      - "127.0.0.1:3001:3001"

  nginx:
    image: nginx:1.27-alpine
    restart: always
    depends_on: [api, web]
    ports:
      - "80:80"
      - "443:443"
    volumes:
      - ./nginx.conf:/etc/nginx/conf.d/default.conf:ro
      - ./letsencrypt:/etc/letsencrypt:ro

  pgdata:

5.4 Nginx reverse proxy

~/erp/nginx.conf — routes api. → API, erp. → web:

server {
  listen 80; server_name api.yourschool.com;
  return 301 https://$host$request_uri;
}
server {
  listen 443 ssl http2; server_name api.yourschool.com;
  ssl_certificate     /etc/letsencrypt/live/api.yourschool.com/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/api.yourschool.com/privkey.pem;
  client_max_body_size 10M;
  location / {
    proxy_pass http://api:3000;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}
server {
  listen 80; server_name erp.yourschool.com;
  return 301 https://$host$request_uri;
}
server {
  listen 443 ssl http2; server_name erp.yourschool.com;
  ssl_certificate     /etc/letsencrypt/live/erp.yourschool.com/fullchain.pem;
  ssl_certificate_key /etc/letsencrypt/live/erp.yourschool.com/privkey.pem;
  location / {
    proxy_pass http://web:3001;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
  }
}

5.5 Issue SSL with certbot

# Stop nginx briefly so port 80 is free
docker compose down
sudo apt install -y certbot
sudo certbot certonly --standalone \
  -d api.yourschool.com -d erp.yourschool.com \
  --agree-tos --email you@yourschool.com --no-eff-email
sudo mv /etc/letsencrypt ~/erp/letsencrypt
docker compose up -d

Renewal: 0 4 * * 0 root certbot renew --quiet && cd ~/erp && docker compose restart nginx

6. CI / CD with GitHub Actions

Path A (App Platform)

Built in. Settings → Auto-deploy on push to main is on by default. To add real CI (lint + tests before deploy): 

# .github/workflows/ci.yml
name: CI
on: [push, pull_request]
jobs:
  backend:
    runs-on: ubuntu-latest
    defaults: { run: { working-directory: backend/school-management-system-backend } }
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: yarn }
      - run: yarn install --frozen-lockfile
      - run: yarn prisma generate
      - run: yarn build
      - run: yarn lint
  frontend:
    runs-on: ubuntu-latest
    defaults: { run: { working-directory: admin_panel/school-management-system-frontend } }
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: 20, cache: yarn }
      - run: yarn install --frozen-lockfile
      - run: yarn build

If CI passes, App Platform auto-deploys. If CI fails, deploy never starts.

Path B (Droplet) — full deploy pipeline

Two stages: build & push Docker images, then SSH to droplet & restart.

# .github/workflows/deploy.yml
name: Deploy
on:
  push:
    branches: [main]
jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions: { contents: read, packages: write }
    steps:
      - uses: actions/checkout@v4
      - uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}
      - name: Build & push API
        uses: docker/build-push-action@v5
        with:
          context: backend/school-management-system-backend
          push: true
          tags: ghcr.io/${{ github.repository_owner }}/school-erp-api:latest
      - name: Build & push Web
        uses: docker/build-push-action@v5
        with:
          context: admin_panel/school-management-system-frontend
          push: true
          tags: ghcr.io/${{ github.repository_owner }}/school-erp-web:latest

  deploy:
    needs: build-and-push
    runs-on: ubuntu-latest
    steps:
      - name: SSH and roll
        uses: appleboy/ssh-action@v1
        with:
          host:     ${{ secrets.DROPLET_HOST }}
          username: ecole
          key:      ${{ secrets.DROPLET_SSH_KEY }}
          script: |
            cd ~/erp
            docker compose pull
            docker compose up -d
            docker compose exec -T api yarn prisma migrate deploy
            docker system prune -f

Add three secrets in GitHub → Settings → Secrets and variables → Actions:

  • DROPLET_HOST — your droplet IP
  • DROPLET_SSH_KEY — paste the private key matching the one on the droplet
  • (GITHUB_TOKEN is provided automatically)

Dockerfiles you need

7. Domain & SSL (Cloudflare)

  1. Add your domain to Cloudflare (free plan).
  2. Update nameservers at your registrar to Cloudflare's. Wait ~10 min.
  3. In Cloudflare DNS:
    • Path A: add a CNAME api → school-erp-api-xxxxx.ondigitalocean.app (proxied, orange cloud). Same for erp.
    • Path B: add A records api & erp → your droplet IP (proxied).
  4. SSL/TLS → Full (strict).
  5. Done. HTTPS auto-issued.

8. Backups

  • Postgres (Path A) — automatic daily backups included with managed DB. 7-day retention by default. Can restore via UI.
  • Postgres (Path B) — set up a nightly pg_dump cron uploading to Spaces:
# /etc/cron.daily/erp-backup.sh
#!/bin/sh
set -e
TS=$(date +%Y%m%d-%H%M)
docker compose -f /home/ecole/erp/docker-compose.yml exec -T postgres \
  pg_dump -U erp_user school_erp | gzip > /tmp/erp-$TS.sql.gz
aws --endpoint=https://blr1.digitaloceanspaces.com s3 cp \
  /tmp/erp-$TS.sql.gz s3://your-backup-bucket/postgres/erp-$TS.sql.gz
rm /tmp/erp-$TS.sql.gz
  • Droplet snapshots — enable automated weekly snapshots ($2.40/mo). Full disk image, one-click restore.
  • Code — already on GitHub.
  • Uploads — if using Cloudinary, they handle it. If using Spaces, enable versioning.

9. Monitoring & alerts

  • Uptime — UptimeRobot (free) pings https://api.yourschool.com/api/health every 5 min, emails you on failure.
  • App logs (Path A) — built-in App Platform Insights. Real-time + 7 days history.
  • App logs (Path B) — docker compose logs -f --tail=100. For long-term, ship to Better Stack (free 1 GB/mo).
  • Error tracking — Sentry (free 5k events/mo). Add @sentry/nextjs on FE and @sentry/node on BE.
  • Health endpoint — already exposed at /api/health. Add Postgres + Redis pings inside it.
  • DB monitoring — managed Postgres has the "Insights" tab (slow queries, connection pool usage).

13. Other ideas worth considering

  • Vercel for the frontend, DO for the API. Vercel's free tier hosts Next.js perfectly. You'd only need a $15/mo Postgres + tiny droplet or App Platform for the API. Cheapest viable production combo.
  • Hetzner Cloud + Coolify — €4/mo for 2 vCPU + 4 GB RAM. Coolify (open-source Heroku) auto-deploys from GitHub. Aggressive value for money but Bangalore latency varies.
  • Neon Postgres serverless — generous free tier, auto-pause, branching. Pair with App Platform or Vercel for near-zero cost on small schools.
  • Cloudflare Tunnel — keep the actual server private (no public IP) and expose only via Cloudflare. Great if you want hybrid (school's on-prem server but accessible from the internet).
  • Multi-tenant on a single deployment — the schema already has schoolId on every model. You can host 10+ schools on the same Path A or B deployment by creating one School row per tenant. Cost stays flat, revenue scales.
  • Read replicas for Postgres when the school grows past 5k students — managed DB gives you a one-click standby for $7/mo.
  • CDN for static assets — already free via Cloudflare proxy. For heavier media, push uploads to Spaces with CDN enabled.
  • Mobile app later — the API is already JWT-based and CORS-aware. A React Native app can hit the same endpoints.
